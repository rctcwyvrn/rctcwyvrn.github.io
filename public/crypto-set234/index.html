intermediate 2 using AES Then intermediate 2 xor c1 = plaintext block</p> <p>The idea is that you can now just try random ints i until the plaintext block ends in \0x01, and the oracle will tell you because it'll be the only attempt that results in valid padding</p> <p>So now you know i XOR last byte of the intermediate = \0x01, so last byte of the intermdiate = \0x01 XOR i, and you just stole a byte of unencrypted intermediate To get the plaintext you just xor it with the original ciphertext as part of the normal CBC proccess</p> <p>Much like earlier challenges, this attack is not theoretically difficult but it is very hard to get working correctly in actuality. Some tips:</p> <ol> <li>Print statements everywhere</li> <li>Start with just the loop for finding one byte, then make it work for 16 bytes, then make it work for all blocks</li> <li>The very first block of the ciphertext is tricky, you can either mess with the IV or make a reordered ciphertext like I did</li> <li>Like the hint says, the fact that the actual plaintext is padded is actually completely irrelevant. Make sure you understand why this is before moving on, it's an important conceptual link.</li> </ol> <p>Pitfalls:</p> <ol> <li>Make sure that you are generating the correct bytes in c1 for the bytes that you already know the intermediate values of</li> <li>Make sure your verify+strip function from set 2 is bugless, it should accept a block that is all padding but reject a block with no padding</li> <li>A general coding tip but you should try to keep your eventual goal in mind as you build up the function</li> <li>You decrypt the bytes in reverse order, so make sure you account for that</li> <li>At the very end remember to XOR the intermediates with the IV + cipher_text</li> </ol> <h2 id="challenge-18-ctr-mode">Challenge 18: CTR Mode</h2> <p>The wikipedia article for CTR mode gives a good explanation, and overall CTR mode is very simple, just don't be dumb like me and get your key and block reversed <a href="https://en.wikipedia.org/wiki/Block_cipher_mode_of_operation#Counter_(CTR)">CTR</a> Also remember to use little endian versions of the nonce and counter: <a href="https://en.wikipedia.org/wiki/Endianness#Little-endian">Endianness</a> , if you don't know it basically means the bytes are stored in reverse order of what we expect; least significant bytes first.</p> <h2 id="challenge-19-20-breaking-a-constant-nonce-ctr">Challenge 19/20: Breaking a constant nonce CTR</h2> <p>I'm not sure what they wanted for challenge 19, I just wrote the code for what I thought they wanted me to do and ended up doing challenge 20... We can then recover x one bit at a time because</p> <p>x_n = y_n xor x_(n-18) n=18...</p> <p>And it turns out this approach works for the other type of shift and also the ones with &amp; so yay</p> <p>The cloning part is simple, just make a constructor that takes a MT state array instead of a seed</p> <h2 id="challenge-24-write-the-mt-stream-cipher-and-break-it">Challenge 24: Write the MT stream cipher and break it</h2> <p>Writing the cipher: straightforward<br /> Breaking it: Not obvious but it turns out that 2^16 keys isn't that many keys, and testing them is very quick. Make sure you understand why the attack works instead of just copying the instructions!</p> <h2 id="challenge-28-implement-a-sha-1-mac">Challenge 28: Implement a SHA-1 MAC</h2> <p>The implementation I used: <a href="https://github.com/ajalt/python-sha1">link</a></p> <p>For the fellow crypto noob, MAC stands for Message Authentication Code. The idea is that the hashing function (sha-1) takes the message+key and generates a hash. The important part of this hash is that small modifications to the resulting hash do not correspond in any way to small changes to the plaintext that was sent, so no bitflipping attacks or anything like that.</p> <p>So in an actual setting:</p> <ol> <li>Alice and Bob share a key</li> <li>Alice tells Bob she is sending message X</li> <li>Alice sends Bob sha-1(key+x)</li> <li>Bob recieves the hash and compares it with his own sha-1(key+x), and if they match then he authenticates her</li> </ol> <p>I'm guessing the messages x would all be something like "auth=____" with user/admin/whatever and Bob would just try them all until one worked, and then give out the proper authority. So to cheat this we need to be able to send fake hashes without knowing the key, which seems impossible at first beause a small change in input results in a completely different hash output.</p> <p>SHA stands for secure hash algorithm, so it's probably safe right?</p> <h2 id="challenge-29-break-sha-1-keyed-mac">Challenge 29: Break SHA-1 Keyed MAC</h2> <p>Oh... "Secret prefix SHA-1 MACs are trivially breakable"</p> <p>I sort of fumbled my way though this one, and the instructions are super not helpful</p> <p>Useful information:</p> <ol> <li>SHA-1 works kinda like the twister from earlier, it has an internal state which it uses to generate it's result. So if we know the state we can generate more, which is why we need to do the same thing as the fake_twister where we could plug in the entire state array and use that as a clone, except this time we use our clone to generate a longer hash which includes our secret sauce</li> </ol> <p>Here's how SHA works (kinda)</p> <ol> <li>Take an input and turn it into bits</li> <li>Append a 1</li> <li>Append 0's until the length is 448</li> <li>The length of the message is converted into 64 bits and added to make a mod 512 == 0 bit message</li> <li>Each 512 bit chunk is then put through the hashing function with an internal state, the state updating for each block</li> <li>At the very end it returns the state variables concatenated together</li> </ol> <p>So since each 512 bit chunk is done seperately, if we can make a sha-1 clone that has the correct state of just finishing the last block of the original message, we can continue on and hash whatever we want</p> <p>So the plan is</p> <ol> <li>Get state from hash, which coresponds to hashing key+message+padding</li> <li>Create a fake sha-1 with that state</li> <li>Ask the fake sha-1 to hash a ;admin=true; token</li> <li>The result turns out to be the same as if you asked it to hash key+message+padding+token, which is exactly what we wanted</li> </ol> <p>I was confused for a while about why we needed to go through the hassle of generating the padding, it's so that we can check to make sure that the hash is what we wanted, and for literally no other reason. We can generate the fake key+message+padding+token but if we wanted to make sure its right we need to actually hash key+message+padding+token</p> <h2 id="challenge-30-break-md4-keyed-mac">Challenge 30: Break MD4 Keyed MAC</h2> <p>Almost identical to challenge 29, but the padding for MD4 uses a little endian representation of the msg length, not big endian like SHA-1</p> <h2 id="challenge-31-artificial-timing-leak-hmac-sha1">Challenge 31: Artificial timing leak HMAC-SHA1</h2> <p>So the main hint for this challenge comes from the aptly named insecure_compare function. How can we exploit the fact that it compares one byte at a time and delays?</p> <p>The idea is that we can just try all the possible first bytes, and for one of them the insecure_compare will find one correct byte and delay once, and for the rest it won't delay at all.</p> <p>The plan:</p> <ol> <li>Guess a byte</li> <li>Make known_signature + guess</li> <li>Pad it out to 40 bytes</li> <li>Send it and record the time taken</li> <li>Floor time_taken / delay to get the number of successful compares that insecure_compare did</li> <li>If that's greater than what we already have then add it on to known and loop</li> </ol> <p>This one takes forever to run on the default 50ms delay though, my solution worked until 20ms delays so I would try running that first</p> <h2 id="challenge-32-less-artificial-timing-leak">Challenge 32: Less artificial timing leak</h2> <p>How to make the attack more consistent? Replace step 5 because the delays are too small to really be consistent, instead just make a list of all the times and take whatever guess caused the longest time. Also running each guess 3 times and taking the average probably helps.</p> <p>My solution worked for 5ms delays but not really for anything less. Maybe taking the 10 longest times and running them all 10 times each, then taking the longest time out of that to be the best?</p>
